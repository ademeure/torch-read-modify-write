# Architecture & Implementation Deep Dive

## Overview

pytorch-auto-graph decomposes PyTorch models into their lowest-level representation (aten ops) and provides tools to capture, verify, visualize, and map those ops to the actual GPU kernels that execute them. The system operates at three levels of the PyTorch stack:

1. **FX level**: High-level ops with module structure preserved (via `torch.compile` custom backend)
2. **Aten level**: Fully decomposed primitive ops with forward+backward (via `aot_autograd`)
3. **Triton/kernel level**: Fused GPU kernels generated by inductor (via an enrichment pass over the captured aten graph)

---

## Module Architecture

```
                    User Entry Points
                    ─────────────────
                    auto.py          (CLI / zero-mod script extraction)
                    __init__.py      (Python API)
                          │
            ┌─────────────┼─────────────────────┐
            ▼             ▼                      ▼
       capture.py    export.py              tensor_dump.py
       (FX graphs)   (aten graphs)         (verification)
            │              │                     │
            │    ┌─────────┼──────────┐          │
            ▼    ▼         ▼          ▼          ▼
      inspector.py   visualizer.py  triton.py   compare_tensors()
      (analysis)     (rendering)    (kernels)   (validation)
            │              │          │
            ▼              ▼          ▼
      editor.py      HTML/JSON      kernel source
      (mutation)                   files
```

### Data Flow

```
Model + Inputs
     │
     ├──→ capture.py ──→ CapturedGraph (FX-level, high-level ops)
     │         │
     │         ├──→ inspector.py ──→ op counts, shapes, dependencies
     │         ├──→ visualizer.py ──→ HTML, JSON
     │         └──→ editor.py ──→ modified GraphModule
     │
     ├──→ export.py ──→ AtenCapture (aten-level, fw+bw, authoritative graph identity)
     │         │
     │         ├──→ Phase 1: FX trace (source annotations)
     │         ├──→ Phase 2: aot_autograd (aten decomposition)
     │         ├──→ Phase 3: export_aten_program (standalone .py)
     │         └──→ triton.py ──→ kernel/debug enrichment
     │                    │
     │                    ├──→ Triton kernel source code
     │                    ├──→ Extern/native kernel calls
     │                    └──→ Aten op → kernel mapping
     │
     └──→ tensor_dump.py ──→ DumpResult (verification)
               │
               ├──→ Reference tensors (all intermediates)
               ├──→ Actual tensors (replay from inputs)
               └──→ Comparison report
```

## Implementation Details

### 1. Graph Capture (`capture.py`)

**How it works**: Registers a custom backend with `torch.compile`. When TorchDynamo traces the model, it passes FX GraphModules to our backend instead of inductor.

**Key design choice**: Deep-copies the GraphModule and example inputs before storing them, because TorchDynamo may mutate or garbage-collect the originals after the backend returns.

**Trade-off**: Deep-copying adds overhead and memory usage, but is necessary for correctness. The `deep_copy=False` option exists for when users want to inspect the graph within the backend callback itself (before it gets mutated).

**Passthrough mode**: The `passthrough_backend` parameter lets you capture the graph AND still compile it with a real backend (e.g. inductor). This is useful for profiling: capture the graph for analysis, but still get the performance benefits of compilation.

```
torch.compile(model, backend=capture.backend)
                              │
                              ▼
              capture.backend(gm, example_inputs)
                    │                    │
                    ▼                    ▼
            deep_copy(gm)         gm.forward (returned)
            store in .graphs      (used by dynamo for execution)
```

### 2. Aten Export (`export.py`)

This is the most complex module. It has three phases:

#### Phase 1: Source Trace Capture

**Problem**: After `aot_autograd` decomposes the model into aten ops, the connection to the original source code is lost. Aten ops like `aten.mm` don't know they came from `self.fc1(x)`.

**Solution**: Run `torch.compile` with a lightweight custom backend first (no aot_autograd). This preserves the FX-level metadata:
- `stack_trace`: Full Python stack trace showing file/line/code
- `source_fn_stack`: Chain of source functions (e.g. `linear → addmm`)
- `nn_module_stack`: Module path (e.g. `self.blocks.0.attn.c_attn`)

We store these in a `source_map` dict keyed by `source_fn` name.

**Trade-off**: This adds an extra compile pass. It's fast (no autograd, no codegen) but doubles the compile time for small models. We could skip it with `annotate_sources=False`.

#### Phase 2: aot_autograd Capture

**Problem**: We need the full forward AND backward computation as explicit aten ops, not hidden behind autograd.

**Solution**: Use `aot_module_simplified` which:
1. Traces the forward through autograd
2. Generates an explicit backward graph
3. Passes both to our `fw_compiler` and `bw_compiler` callbacks

**FakeTensor handling**: `aot_autograd` runs under FakeTensorMode - tensors have shapes and dtypes but no data. Our `_safe_copy_inputs` detects FakeTensors by class name and creates real random tensors with matching shapes. This was a critical bug fix: the original code used `inp.clone().detach()` which silently succeeds on FakeTensors but produces another FakeTensor.

**Primal mapping**: The aten graph's inputs are anonymous (`primals_1`, `primals_2`, ...). `_build_primal_map` resolves these to human-readable names (`self.fc1.weight`, `input [2, 8]`) using shape matching against the model's parameter list. When shapes are ambiguous (e.g. two layers with the same weight shape), it uses consumer metadata (which module the primal feeds into) for disambiguation.

#### Phase 3: Python Export

**Problem**: Generate a standalone `.py` file that can be edited and rerun without the original model code.

**Solution**: Walk the FX graph node by node, converting each to a Python statement:
- `call_function` nodes → `torch.ops.aten.op_name(args...)`
- `placeholder` nodes → function parameters with type annotations
- `output` nodes → return statement

**Source annotations**: Cross-reference each aten op's `source_fn_stack` with the Phase 1 source map to insert comment headers showing which PyTorch module/line each group of ops came from.

**Aten op naming**: We strip the `.default` overload suffix (it's implicit) and use the `aten.` shorthand prefix instead of `torch.ops.aten.`. Named overloads like `.Tensor` or `.int` are preserved.

**Weight embedding**: Small tensors (< `inline_threshold` elements) are inlined as `torch.tensor([...])`. Large tensors are saved to a `.pt` file and loaded at runtime. The threshold is tunable.

### 3. Inductor Debug Enrichment (`triton.py`)

**How it works**: After the aten graph has already been captured, we optionally compile the same model and inputs with `torch.compile(backend="inductor")` and `trace.enabled=True`. Inductor generates Triton kernel source code and writes debug artifacts to a cache directory. We then parse those artifacts and attach the resulting kernel metadata back onto the captured aten graph.

**Cache clearing**: The inductor cache (`/tmp/torchinductor_root/`) is cleared before each enrichment pass. Without this, subsequent runs reuse cached kernels and don't regenerate the debug artifacts we need. This is a conscious trade-off: we sacrifice compilation speed for reliable artifact generation.

**Parsing strategy**: The inductor output code (`output_code.py`) has a structured format with standardized comments:

```python
# Topologically Sorted Source Nodes: [linear, x], Original ATen: [aten.addmm, aten.relu]
# Source node to ATen node mapping:
#   linear => add_tensor
#   x => relu
triton_poi_fused_addmm_relu_0 = async_compile.triton('...', '''
    @triton.jit
    def triton_poi_fused_addmm_relu_0(...):
        ...
''')
```

We parse this with regex to extract:
1. **Kernel definitions**: Triton `@triton.jit` functions with their source code
2. **Aten op mapping**: Which aten ops are fused into each kernel
3. **Extern kernels**: cuBLAS calls (detected as `extern_kernels.mm/addmm/bmm`)
4. **Source node mapping**: Which high-level PyTorch ops map to which aten ops

**Kernel types**:
- `triton`: Fused pointwise/reduction kernels generated by inductor. These are the interesting ones - they show which ops get fused together.
- `extern`: Library calls (cuBLAS matmul, cuDNN conv). These aren't generated by inductor but dispatched to optimized vendor libraries.

**NanoGPT example**: 12 Triton kernels + 13 extern (cuBLAS) calls. Key fusions:
- LayerNorm + embedding + add → single Triton kernel
- Softmax (prepare + sub + exp) → single Triton kernel
- GELU → single Triton kernel
- All matmuls → cuBLAS extern calls (not fused with anything)

### 4. Tensor Dumping & Verification (`tensor_dump.py`)

**Core idea**: If a computation graph is correct and deterministic, running it twice with the same inputs must produce identical intermediate tensors at every node.

**Implementation**: Uses `torch.fx.interpreter.Interpreter` which executes a GraphModule node by node (instead of all at once). Our `RecordingInterpreter` subclass intercepts `run_node()` to clone and save every tensor after each op.

**Input reconstruction** (`_build_real_forward_inputs`):

The hardest problem in this module. `aot_autograd` flattens model parameters and user inputs into a single list of "primals", but the ordering is not simply [all_params, all_inputs]. It depends on the parameter access order during tracing.

Our solution: Shape-based matching. For each graph placeholder, we check its shape/dtype (from FakeTensor metadata) against unused model parameters. If it matches a parameter, we use the real parameter value. If no parameter matches, it must be a user input.

```
Graph placeholders:  primals_1[16,8]  primals_2[16]  primals_3[2,8]  primals_4[4,16]  primals_5[4]
Model params:        fc1.weight[16,8] fc1.bias[16]   fc2.weight[4,16] fc2.bias[4]
User inputs:         x[2,8]

Shape matching:
  primals_1[16,8] → fc1.weight ✓
  primals_2[16]   → fc1.bias ✓
  primals_3[2,8]  → (no param match) → user input x ✓
  primals_4[4,16] → fc2.weight ✓
  primals_5[4]    → fc2.bias ✓
```

**Backward inputs**: The backward graph's inputs are saved tensors from the forward output tuple plus gradient tensors. These may be FakeTensors. We materialize them to real random tensors matching the expected shapes. The backward determinism check still works because both runs use the same materialized inputs.

**Known limitation**: The Conv+BN backward produces NaN mismatches. This is because `native_batch_norm_backward` uses running statistics that may produce NaN when the forward used random (not statistically meaningful) saved tensors. The forward pass always matches perfectly.

### 5. Auto Extraction (`auto.py`)

**Problem**: Users have arbitrary training scripts. We want to extract graphs without modifying their code.

**Solution**: Monkey-patch `nn.Module.__call__`. When installed, the `_ModelInterceptor` records every `nn.Module` that gets called during script execution, along with a snapshot of its inputs. After the script runs, we process each captured model through the full aten export pipeline.

**Heuristics**:
- Only captures models with >= `min_params` parameters (default 100) to skip trivial modules
- Only captures the first call to each model class (skips repeated iterations)
- Caps at `max_captures` distinct models (default 10)
- Catches `SystemExit` and other exceptions gracefully

**Input inference** (`_infer_example_inputs`): When extracting from a module file without example inputs, we inspect the `forward()` signature and model structure to guess appropriate inputs:
- Parameters named `idx`/`token`/`input_id` → `torch.randint(0, vocab_size, (1, 16))`
- Parameters named `mask` → `torch.ones(1, 16, dtype=torch.bool)`
- Default → float tensor matching first layer's `in_features`

### 6. Inspector (`inspector.py`)

Provides structured analysis of FX graphs: op counts, categorization (arithmetic/activation/normalization/etc.), shape tables, dependency chains, and node search. Uses keyword matching for categorization.

### 7. Editor (`editor.py`)

Programmatic graph editing with undo. Supports replace, insert, remove, rewire, and fuse operations. Each edit is checkpointed for undo. After editing, `compile()` recompiles the graph into a runnable `GraphModule`.

### 8. Visualizer (`visualizer.py`)

Multi-format rendering:
- **HTML**: Interactive canvas-based viewer with zoom/pan/search, node inspection sidebar, source code group highlighting, parameter collapsing. ~1000 lines of embedded JS.
- **JSON**: Machine-readable export.

## Key Design Decisions & Trade-offs

### 1. Authoritative Graph Capture, Optional Kernel Enrichment

**Decision**: Treat the aten capture path as the source of truth, then optionally run a second Inductor compilation to enrich that capture with kernel-level lowering details.

**Why**: `aot_autograd` strips source metadata. The FX-level trace preserves `stack_trace`, `source_fn_stack`, and `nn_module_stack` that we need for source annotations.

**Trade-off**: 2x compile time for source-annotated output vs no annotations. Acceptable because both passes are fast (no codegen).

**Alternative considered**: Extracting source info from the aten graph's metadata directly. But `aot_autograd` propagates only partial metadata to the aten graph (mainly shapes/dtypes, not source traces).

### 2. Shape-Based Primal Matching

**Decision**: Match graph primals to model parameters by tensor shape.

**Why**: `aot_autograd` doesn't preserve the mapping from primals to parameter names. The ordering of primals depends on the tracing order, which isn't guaranteed to match `model.named_parameters()`.

**Trade-off**: Fails when multiple parameters have the same shape AND dtype. Mitigated by consumer-based disambiguation (checking which module the primal feeds into).

**Alternative considered**: Instrumenting `aot_autograd` to record the mapping directly. Would require monkey-patching PyTorch internals.

### 3. FakeTensor Materialization

**Decision**: Detect FakeTensors by class name and replace with real random tensors.

**Why**: `aot_autograd` runs under FakeTensorMode. Example inputs captured during compilation are FakeTensors that can't be used for actual computation.

**Trade-off**: Random tensors don't match the model's real weights, so the computed values are meaningless in absolute terms. But for determinism verification, this is fine: two runs with the same random inputs produce the same outputs.

**Alternative considered**: Capturing real tensors during the forward pass (before aot_autograd). This would require hooking into the model's actual execution, which `_build_real_forward_inputs` now does by pulling directly from the model's state dict.

### 4. Inductor Cache Clearing for Triton Capture

**Decision**: Clear the entire inductor cache before capturing Triton kernels.

**Why**: Inductor only writes debug artifacts (output_code.py, etc.) when compiling from scratch. Cached compilations skip artifact generation.

**Trade-off**: Slow (forces full recompilation every time). Could be improved by tracking cache keys or checking for existing artifacts.

### 5. Regex-Based Triton Output Parsing

**Decision**: Parse the inductor-generated `output_code.py` with regex patterns.

**Why**: The output code has a stable, structured format with standardized comments. It's simpler and more maintainable than hooking into inductor's internal APIs.

**Trade-off**: Brittle if inductor changes its output format. But the format has been stable across PyTorch versions. If it breaks, the regex patterns are localized and easy to update.

**Alternative considered**: Using inductor's internal `CodegenSchedulerNode` API. Would be more robust but tightly couples to PyTorch internals.
